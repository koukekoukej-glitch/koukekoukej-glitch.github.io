透明执行

赋予 Agent 远程操作用户电脑的能力

Agent 跑在服务器上，但用户工作区在本地电脑，且两边的文件范围不一样，要尽可能保证Agent在远程执行的时候也有操作本地文件系统一样的效率

在 Agent 和文件系统之间有一层透明代理——Agent 调 SDK 标准工具时，系统自动判断路径该本地处理还是转发到用户电脑上的桌面客户端执行，Agent 完全不知道自己在远程操作

每次会话开始时，做一次目录检测：两边都有的配表和前端代码优先读用户本地，保证和用户一致、只在服务器的后端代码直接读服务器
Agent 看到的是一个合并后的统一文件系统，不需要知道文件来自哪台机器

写入自动对接 Perforce 工作流：首次写入时惰性创建专属 Changelist，后续写入归到同一个 CL，策划在 P4 客户端里验收后再提交
断线时读操作回退到服务器副本，写操作直接停止，不中断活跃会话

上下文工程

教会 Agent 项目的私域知识

Agent 所需的上下文分为两个正交的维度 —— 能力 和 领域知识
两个维度的变化驱动力完全不同：新增一种能力不影响装备系统的规则，新增一个游戏模块不影响配表流程

Skills 能力

每个 Skill 描述完整的工作流程和判断规则，不依赖外部知识："词缀配置专家"精通词缀的标准流程，"系统调查助手"知道如何追踪代码
用户也可以封装自己的 Skill 发布给团队，订阅者实时获得更新

Docs 领域知识

250+ 个按游戏系统组织的知识库，装备、词缀、赛季各模块独立，被 Skill 按需加载
用一篇描述了所有系统的核心术语和职责范围的文档作为入口索引，确保 Agent 能够在遇到相关问题时读取正确的文档，同时维护一份描述系统间耦合关系的数据库，Agent 阅读相关文档时直接注入该系统的耦合关系查询结果，让 Agent 能更好地解决横跨多个系统的任务

数据驱动

让真实使用反过来塑造 Agent

Agent 上线后，还需要持续根据线上数据监控 Agent 表现——一个失败的回答到底是缺了领域知识、方法论不完备、还是用户问题本身没描述清，需要更深的归因
锐评系统打通"用户反馈 → 自动诊断 → 上下文优化"的闭环，让每一次真实对话都能反过来推动 Agent 改进

用户锐评

每个会话界面里有一个"锐评"按钮，用户对 Agent 表现不满时可以一键提交，或额外附加自由文本说明
用户在反馈队列页面可以追踪自己提交的反馈被分析到哪一步、看到最终的诊断结果

自动诊断

系统自动导出完整会话——消息、工具调用链、Agent 的中间推理——并合并工具调用次数、耗时、费用、错误分类等量化指标
启动一个独立的 Claude 实例，加载专门的 session-optimizer Skill，对照项目 CLAUDE.md 做第一视角还原，定位是上下文工程的哪一环出了问题，产出可执行的修改方案
分析在隔离的 git worktree 里运行，分析器的上下文被精简到只保留分析所需信息，避免被项目噪音污染结论

闭环验证

修改方案里要求写下可证伪的预期：用具体指标描述修复后会发生什么变化
方案应用后重跑原会话——克隆工作区做完全隔离，把多轮对话智能压缩成一个综合 prompt 跑在新配置上，重跑后的最终结果、工具调用统计、错误数与预期逐一比对，确认修复有效后再合并回线上

项目专属工具

让 Agent 处理项目中的复杂格式文件

游戏项目的核心数据不是代码和文本文件——配置表是 Excel，引擎资产是二进制格式，数据表之间有大量相互引用关系
Agent 原生的文件读写能力处理不了这些，需要专用工具

Field Graph

策划最常问的问题是"改了这个字段会影响哪些地方？"，项目中有 800+ 张配置表，彼此通过外键大量引用，但没有标准 Schema
Field Graph 为所有配置表预建了一份关系索引，覆盖字段间的外键关系、各代码层对字段的引用、以及跨表的数据依赖
Agent 通过它可以：查询任意字段的结构定义和数据来源、按条件筛选记录、沿外键链逐层追踪影响范围、跨全部配置表的全文搜索、以及数据完整性校验，每个字段都标注了来源信息，让Agent直接知道该去哪里修改

Table Writer

Excel 文件的内部结构比看起来复杂得多，直接用通用库写入容易破坏公式引用和格式信息
Table Writer 封装了安全的写入方式：优先调用 Excel 自身的保存机制保证格式零损坏，在 Excel 不可用时回退到精确的定点修改——只动被改的单元格，其余内容保持不变
写入前自动完成版本控制的 checkout 流程，写入后可触发数据校验和导出，Agent 如果试图绕过它直接写 Excel 文件，会被 Hook 拦截并引导回来

UE HTTP API

引擎内部的配置数据和美术资产是二进制格式，外部程序无法直接安全修改
我们在引擎编辑器内部署了一个 HTTP 服务插件，Agent 通过它可以读写引擎配置表的行数据、创建和编辑美术资产、生成动画相关的蓝图文件、以及触发数据导出
所有操作在引擎内部原子完成——修改、保存、导出在一次调用中搞定，避免了外部工具和引擎编辑器争抢文件锁的问题

外部工具集成

连接团队已有的工作平台

Agent 不只操作项目代码和配置，还能直接读写团队日常使用的协作平台，让用户无需在多个工具之间来回粘贴上下文

即时通讯

接入 Slack 和飞书，用户在群里 @ 即可使用 Agent 能力，不需要打开独立的网页界面
飞书侧接入更深：除了对话，还对接了工作流引擎，通过事件驱动 Agent 自动执行配置工作和多维表格读写

项目管理

接入 Jira，Agent 能查任务、找负责人、跟进度、看排期，帮助用户在对话中直接获取项目管理信息

策划文档

项目的历史文档都在 Confluence 上，因维护不够完善，没有作为默认上下文使用；但用户需要时可以主动触发检索：读取时自动把 Confluence 的富文本格式转成 Markdown 供 Agent 理解，也能把工作成果直接写回源页面

数据查询

接入线上数据仓库，用户用自然语言描述需求，Agent 生成 SQL 查询并返回结果：对比不同版本的首日付费收入、分析新版本玩家留存和行为路径、查询特定玩法的参与率和收益数据等
Agent 在执行前会先和用户确认查询条件和数据源，避免查错表或遗漏过滤条件

服务端诊断

每个团队成员都有自己的个人测试服，用户排查问题时，Agent 可以直接访问服务端错误日志、读取运行时配置文件、查询游戏数据库，不需要找研发帮忙
